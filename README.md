# [專題名稱：HabitFlow - 跨平台習慣追蹤利器]

## Proposal Report

### 動機與目標
作為一名大學生，我深知建立良好生活習慣（如規律運動、閱讀、背單字）的挑戰性。市面上雖有許多 Habit Tracker App，但大多數優秀的功能（如建立超過 3 個習慣、詳細統計圖表）都需要昂貴的訂閱費用。

本專題的目標是利用 **Flutter** 開發一款專為學生設計、完全免費且開源的習慣追蹤 APP。透過直觀的 UI 與科學的數據回饋，幫助使用者建立自律生活。對我個人而言，這也是將 **Dart 語言**與**資料結構**結合，實踐於移動端開發的最佳練習。

### 預期功能
1.  **習慣管理系統**：支援新增、編輯、刪除習慣，並可設定每日目標次數與自定義標籤。
2.  **互動式打卡介面**：首頁提供快速打卡功能，並即時更新當日進度條。
3.  **連續天數 (Streak) 計算**：自動計算並顯示各項習慣的連續達成天數，激勵使用者不中斷紀錄。
4.  **歷史紀錄與統計**：提供週視圖與月視圖，將打卡數據視覺化（折線圖或熱點圖）。
5.  **本地持久化儲存**：即使在無網路環境下，也能透過本地資料庫確保資料不遺失。

### 使用技術
* **程式語言**：Dart
* **開發框架**：Flutter
* **本地資料庫**：SQLite (sqflite 套件) 或 Hive (NoSQL)
* **狀態管理**：Provider (用於 UI 與資料同步)
* **圖表工具**：fl_chart 套件
* **版本控制**：Git & GitHub

### 與課程的關聯
1.  **資料結構 (Data Structures)**：
    * **List**：用於管理首頁習慣列表與歷史打卡紀錄。
    * **Map**：實作「打卡查詢優化」，將 `DateTime` 映射至 `CheckInStatus`，實現 $O(1)$ 的查詢效率。
2.  **演算法 (Algorithms)**：
    * **Sorting**：使用排序演算法（如內建的 TimSort 概念）根據優先級或時間對習慣進行動態排列。
    * **Time Complexity**：針對大量歷史數據的統計運算進行時間複雜度評估，避免在 UI 線程產生阻塞。
3.  **進階程式設計概念**：
    * **物件導向 (OOP)**：運用封裝與類別繼承來設計 Habit 與 Record 模型。
    * **非同步程式設計 (Async/Await)**：處理資料庫 I/O 操作，確保 APP 介面在讀取資料時不會卡頓。
